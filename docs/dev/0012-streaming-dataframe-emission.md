# 0012 — Формирование датафреймов над непрерывным потоком

## Статус

**Исследование / проектирование.** Cross-cutting: трогает sink, агрегацию и
**storage-ветку (ING-4)**; результат *потребляется* доставкой ([0011](0011-remote-sync.md)),
но это не задача доставки. Выделено из 0011 по правилу серии «один документ —
одно направление» (см. [0009](0009-modularization-granularity.md) про критерий
выноса cross-cut). К одной ветке не привязан.

## Контекст

Источник (SMB-шара/каталог) пополняется **постоянно**, приложение работает в
stream. При таком сценарии понятие «полного набора» отсутствует: барьер «дождаться
полноты» либо **никогда** не сработает (поток не иссякает), либо сработает на
**полупустом** наборе (между бурстами ledger опустеет на миг, пока файлы ещё
доезжают). Вопрос не про доставку — доставка увозит то, что сформировано; больно
именно на границе «поток → файл-датафрейм».

Текущая модель усугубляет: sink/агрегация **перезаписывают** canonical-файл
целиком (keep-first + стабильные id). Под бесконечным стримом это неограниченный
рост файла + перезапись всего на каждом прогоне (фактически O(n²) за жизнь потока)
и неограниченный рост состояния дедупа/id. Снять консистентную копию «на лету» с
мутирующего файла нельзя (torn read).

Опираемся на индустриальную модель стрим-обработки:

- **Dataflow-модель** (Akidau и др., VLDB 2015; книга *«Streaming Systems»*,
  O'Reilly; реализация — **Apache Beam**): любой emit над неограниченным потоком =
  *what* (что считаем) / *where* — **окно** в event-time / *when* — **триггер +
  watermark** / *how* — accumulating vs discarding.
- **Output modes** Spark Structured Streaming: `complete` / `append` / `update` —
  система **заставляет** явно выбрать форму выхода.
- Материализованное представление из потока: Kleppmann, **DDIA** гл. 11; доклад
  *«Turning the database inside out»*.
- Атомарный commit с маркером готовности (`_SUCCESS`) — Hadoop/Spark
  `FileOutputCommitter`; публикация immutable-файлов в data lake.
- Kappa-архитектура (Jay Kreps) — если когда-нибудь захочется и инкременты, и
  периодический полный список.

## Решения

**1. Отказ от «полного набора» → полнота = покрытие на момент watermark.** Не
ждём «когда всё придёт», а **эмитим по триггеру** и **штампуем покрытие** (срез
как-of T). «Полный, точка» в стриме не определён; «полный на момент T» — определён
и полезен.

**2. Выход — неизменяемые атомарные снимки/инкременты, не мутирующий файл.**
- запись через temp → atomic rename;
- immutable-имена с таймстемпом (как уже выбрано в 0011);
- **маркер готовности** рядом со срезом (`_SUCCESS`-стиль): потребитель забирает,
  только если маркер есть;
- публикуемая единица — **каталог immutable-файлов**, а не один мутирующий файл.

Это закрывает идею «копировать файл, пока в него пишут»: цель верная (периодический
частичный срез — осознанное решение), но механизм заменён, чтобы разница между
*defined-partial* и *torn* была на нашей стороне.

**3. Output-mode — явный конфиг первого класса** (словарь Spark):
- `complete` — периодический **полный снимок** текущего набора (удобно
  потребителю «дай весь актуальный список», платим переписыванием всего);
- `append` — **инкремент** по партиции/файлу (дёшево, не блокирует, идеально под
  «файлы капают на шару»; потребитель аккумулирует).

`per-partition` из 0011 — это и есть `append`. Доставка ([0011]) **агностична к
режиму**: увозит то, что сформировано.

**4. Триггер emit — явный, не «барьер по ledger».** Кандидаты: processing-time
interval / per-file / session (quiet-period). Барьер «ledger опустел» как
батч-мышление **отвергнут** — на стриме ломается (см. Контекст).

**5. Дешёвый консистентный снимок — со стороны хранилища.** Формирование под
стримом = ограниченный **UPSERT** в хранилище; снимок = `SELECT → экспорт` на
**консистентном чтении** (SQLite **WAL**: читатель видит консистентный снимок
одновременно с идущими записями). Это и есть «копия во время записи» — но без
torn-read. **Контракт** (immutable-снимок + watermark + маркер + output-mode)
**storage-агностичен и фиксируется здесь**; дешёвая консистентная реализация —
в storage-ветке (ING-4).

## Следствия

- Меняется контракт sink/aggregation: от «перезаписать один canonical-файл» к
  «эмитить immutable снимок/инкремент по триггеру + маркер готовности».
- Новый конфиг-блок (предположительно `ioc.emission` или в составе `aggregation`):
  output-mode, триггер, watermark/именование, маркер.
- Жёсткая завязка на storage-ветку (ING-4): состояние дедупа/id и дешёвый снимок —
  там; этот документ фиксирует контракт, который storage-ветка реализует.
- Доставка ([0011]) читает output-mode, но к нему агностична.
- Возможный новый сигнал в `ioc health`/диагностике: возраст последнего снимка /
  лаг watermark.
- Ретеншен ([maintenance reaper](0001-streaming-ingestion.md)) распространяется на
  накопление immutable-снимков (max-age/max-count).

## Открытые вопросы

1. **Дефолтный output-mode = модель потребителя.** Потребители блок-листов хотят
   «весь актуальный список одним файлом» (firewall/proxy перечитывает целиком →
   `complete`/снимок) или умеют аккумулировать инкременты (→ `append`, дешевле, не
   блокирует)? Встречный вопрос задан, ждём ответа — от него дефолт.
2. **Дефолтный триггер.** Чисто непрерывный поток → processing-time interval;
   бурстовый источник (на деле многие «непрерывные» источники бурстовые) →
   session/quiet-period. Поддержать оба конфигурируемо; какой дефолт.
3. **Дедуп/стабильные id под `append`.** Инкремент не видит глобального состояния
   — как назначать стабильные id и резолвить дубли между инкрементами без полного
   состояния (тянет к storage/индексу id из агрегации).
4. **Поздние данные и поправки** (accumulating vs discarding): нужно ли пересылать
   обновления уже отправленных снимков, или каждый снимок самодостаточен и свежий
   побеждает.
5. **Ретеншен снимков** локально и на приёмнике: чистка существующим reaper'ом или
   отдельная политика; сколько срезов хранить.
