# Сквозные подсистемы

Логирование, диагностика, проброс и трансляция ошибок — это **не утилитарный
фон, а самостоятельная бизнес-логика приложения**. Её потребляют разные сервисы,
поэтому она должна жить отдельно, за своими границами, выделяться в агностичные
модули и работать через порты. Жёстко зашивать зависимости между сервисами и
такими подсистемами — запрещено.

> Статус: **проектирование**. Ниже — целевая модель и порты. Реализация —
> по мере выделения модулей ([modularization.md](modularization.md)).

## Общее правило

- Каждая сквозная подсистема определяет **порты в ядре** (агностично) и
  **адаптеры на инфраструктуре** (SLF4J/Logback, файлы, внешние коллекторы).
- Сервисы зависят на **порт подсистемы**, а не на конкретную реализацию и не
  друг на друга.
- Подсистема не знает о доменной специфике потребителя; потребитель не знает о
  технологии подсистемы.

## Диагностика / телеметрия

Отдельный порт для структурированной диагностики хода работы (что прочитано,
сколько извлечено, что отсеяно, какие стадии отработали). Не путать с
«логированием для разработчика» — это наблюдаемость процесса.

```
port (core):     DiagnosticsSink / TelemetrySink   — приём событий обработки
adapter (infra): LoggingTelemetrySink              — структурированный лог (SLF4J)
                 (в будущем: метрики, JSON-события, внешний коллектор)
```

Принцип: стадии конвейера эмитят доменно-нейтральные события в порт; куда они
уходят (лог, метрика, файл) — решает адаптер и конфигурация.

## Модель и трансляция ошибок

- **Иерархия по слоям.** Доменные нарушения, прикладные ошибки сценария и сбои
  адаптеров — разные категории. Доменное ядро не выбрасывает и не ловит
  инфраструктурные исключения.
- **Трансляция на границах.** Адаптер переводит низкоуровневый сбой (I/O, парсер,
  CSV) в ошибку своего контракта; прикладной слой — в результат сценария. Утечки
  адаптерных исключений в домен недопустимы.
- **Обогащение контекстом.** Ошибка несёт диагностический контекст (источник,
  стадия, индикатор) для воспроизводимости — но без зависимости домена на
  подсистему логирования.

```
port (core):     ErrorTranslator / FailurePolicy   — категоризация и политика
                 (fail-fast | collect-and-continue) обработки сбоев по элементам
adapter (infra): реализации политики + интеграция с диагностикой
```

Пример потребности: при пакетной обработке один «битый» индикатор не должен
ронять весь прогон — политика `collect-and-continue` собирает ошибки и отдаёт их
в отчёт через порт диагностики.

## Логирование для разработчика

SLF4J как фасад, Logback как реализация — деталь инфраструктуры. Доменные и
прикладные классы используют SLF4J-логгер как тонкий инструмент, но **значимые
для процесса** события идут не в «просто лог», а через порт диагностики (см.
выше), чтобы их можно было перенаправить/агрегировать.

## Почему это важно

Вынесение сквозных подсистем за порты даёт то же, что и для основной логики:
заменяемость реализации, тестируемость (подмена порта в тестах), отсутствие
скрытой связности между сервисами и возможность переиспользовать подсистему в
других модулях/приложениях ([modularization.md](modularization.md):
`platform/ioc-platform-logging`, `platform/ioc-platform-errors`).
