# Извлечение, нормализация и тест-корпус

Дизайн извлечения индикаторов (паттерны), нормализации значений и тестового
покрытия. Классификация масок (`url_match`/`host_match`) — в
[output-mapping.md](output-mapping.md#декларативная-классификация-масок-matchurl--matchhost).

> Статус: **дизайн**. Паттерны и нормализация декларативны (`ioc.*`), RE2-safe
> (`\b`, без look-around) — работают на обоих движках `PatternEngine`.

## Покрытие типов

Сейчас: текущие сетевые/файловые + **.onion** + **telegram** (есть в корпусе).
email/IPv6 — позже (в текущих данных отсутствуют; набор расширяется конфигом).

| Тип | Что ловим | Примечание |
|---|---|---|
| `IPV4` | `1.2.3.4` (+ `:port`/`/path`) | порт — только в authority |
| `DOMAIN` | FQDN (+ `/path`) | классификация по PSL |
| `URL` | `https?://…` | хост извлекается для `host_match` |
| `MD5/SHA1/SHA256` | hex 32/40/64 | upper-case в артефакте |
| `.onion` | base32-адрес + `.onion` | спец-TLD → вариант 1 |
| telegram | `t.me/…`, `telegram.me/…`, `api.telegram.org/bot<id>:<token>` | URL; `:token` — не порт |

Приоритет паттернов (claim диапазонов) ставит URL/onion/telegram/IP перед общим
`DOMAIN`, чтобы хост URL не переэмитился как отдельный домен.

## Нормализация

Владельцы — доменные сервисы `IndicatorNormalizer` (нормализует значение) и
`IndicatorFeatureExtractor` (строит `IndicatorFeatures`); `MatchPolicy` работает
над признаками, а не парсит строку в маппере. PSL-признак — через порт
`HostClassifier` (адаптер Guava). Правила:

- Обрезка хвостовой пунктуации (`;`, `,`, кавычки) — частый артефакт текста.
- `:` считаем **портом только в authority** (до первого `/`); иначе это часть
  пути (напр. `api.telegram.org/bot<id>:<token>`).
- Извлечение host из URL для `host_match`.
- Регистр: только **хост** в нижний (`lower-host`) — путь/query/токен сохраняют
  регистр (он чувствителен); хэши — upper. Через `transform`, см. output-mapping.
- IDN/punycode — позже.
- onion-адрес — спец-TLD; PSL может не принять как суффикс → трактуем как
  регистрируемый (вариант 1).

## Классификация (нюансы PSL)

4-вариантное rule-based правило на **Public Suffix List** (Guava, **incl.
private-секция**):

- `duckdns.org`, `workers.dev`, `ngrok.io`, `portmap.host` — в PSL private →
  `x.duckdns.org` регистрируемый → **вариант 1**.
- Провайдеров вне PSL (`tw1.ru`, `yzz.me`, `anondns.net`) PSL не знает →
  `cs371620.tw1.ru` станет **вариантом 2** (в ручном эталоне был 1).
- onion-адрес → **вариант 1**.

## Тест-корпус и изоляция тестов

**Принцип (жёстко): тесты не опираются на артефакты проекта.**

- **Никаких** `source/*`, `dataframe/*`, прод-`application.yml` в тестах. Иначе
  смена источника, значения поля артефакта или одного поля конфига роняет тест.
- **Тест владеет данными.** Фикстуры синтетические, минимальные, создаются в
  `@TempDir` / `src/test/resources` (тестовый контур, tmp). Конфиг/политику тест
  задаёт сам (конструирует доменные объекты или тест-конфиг).
- **Выход sink** — во временные каталоги (`@TempDir`), не в `dataframe/`.
- **Проверяем логику, не текущие данные.**

**Формат — case-table + golden:**

- **Case-table** — декларативная таблица кейсов (в тест-ресурсах):
  `вход → { тип, значение, url_match, host_match }`. Параметризованные тесты
  (JUnit 5 `@MethodSource`/CSV) на экстрактор/нормализацию/классификатор.
  Покрывает все бакеты и краевые случаи.
- **Golden e2e** — маленькая курируемая **синтетическая** вход-фикстура →
  ожидаемый артефакт, сравнение diff. В тест-ресурсах, **не** реальный dataframe.
- Корпус версионируемый и расширяемый; кейсы изолированы — добавление одного не
  ломает остальные.

**Бакеты для покрытия (из анализа эталона + edge):**

| Бакет | Ожидаемо | Пример |
|---|---|---|
| registrable domain | вариант 1 | `zeccecard.com` |
| subdomain | вариант 2 | `zzzzjm2.mlcrosoft.asia` |
| bare IP | вариант 1 | `159.198.41.140` |
| IP + port/path | вариант 3 | `85.209.156.3:8082/wagent.exe` |
| host + path | вариант 3 | `zipurl.link/r5S93Q` |
| host + query | вариант 4 | `…/0201787911?asdzq` |
| .onion | вариант 1 | `pac…onion` |
| PSL private (duckdns/workers) | вариант 1 | `x.duckdns.org` |
| не-PSL провайдер | вариант 2 | `cs371620.tw1.ru` |
| telegram bot URL | вариант 3 | `api.telegram.org/bot…:token` |
| edge: trailing `;` | нормализуется | `voffice.top;` |

## Расхождение с ручным эталоном

Под авторитетное правило **≈20% строк эталона (≈37/185)** сменят вариант на 2/4,
которых в ручной заливке нет. Это ожидаемо: корпус строим под правило, ручной
эталон — **не оракул** для вариантов 2/4.

## Этапы

1. Паттерны `.onion`/telegram + нормализация (authority-port, trailing punct,
   host-extraction).
2. Case-table корпус (бакеты + edge) + параметризованные тесты.
3. Rule-based 4-вариантный `MatchPolicy` на PSL (Guava) + onion/не-PSL.
4. Golden e2e фикстура (синтетическая, в тест-ресурсах).
