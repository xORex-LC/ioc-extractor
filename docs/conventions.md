# Конвенции

Правила оформления кода и документации. Обязательны к соблюдению.

## Javadoc

Качественный Javadoc — обязателен, не опционален.

**Что документируем всегда:**
- Все публичные типы и публичные/protected методы.
- **Порты** документируются как контракты: что гарантирует реализация, каковы
  пред-/постусловия, что считается ошибкой.
- Доменные value objects: смысл полей, инварианты, правила равенства/дедупликации.

**Как пишем:**
- Class-level — назначение и роль в слое (одно-два предложения), при
  необходимости «почему так», а не только «что».
- Методы — `@param` / `@return` / `@throws` там, где это несёт смысл.
- Объясняем неочевидное (напр. почему RE2/J и ограничение на `\b`-паттерны,
  почему рефанг — упорядоченные литеральные замены).
- Без «шумного» Javadoc, дублирующего сигнатуру (`/** Gets name. */`).
- Язык Javadoc — **английский** (как и весь код/идентификаторы).

**Пример (контракт порта):**
```java
/**
 * Secondary (driven) port: extract plain text from a source document,
 * regardless of its format. Implementations must not leak parser-specific
 * exceptions; failures are reported as {@link IocExtractorException}.
 */
public interface SourceReader {
    String readText(Path source);
}
```

> На уровне пакета смысл фиксируем в `package-info.java` (Javadoc пакета)
> дополнительно к README каталога — это разные аудитории: API-доки vs навигация.

## README в каждом каталоге

Каждый значимый каталог (слой, пакет, подсистема, каталог данных) сопровождается
своим `README.md` с краткой информацией о содержимом. README обновляется вместе с
содержимым каталога.

**Шаблон:**
```markdown
# <путь/каталога>

## Назначение

Краткое (1–3 предложения) описание ответственности каталога.

**Правило слоя:** что этому слою/пакету разрешено и запрещено
(направление зависимостей, кто на кого может ссылаться).

## Структура

| Подпапка / файл | Назначение |
|---|---|
| `foo/` | … |
| `Bar.java` | … |

## Зависимости

**Зависит от:** перечень нижних слоёв/модулей.
**Не импортируется:** кем (для внутренних слоёв — кем извне нельзя).

## Точка входа / Заметки

Точка входа, ключевые порты, особенности (опционально).
```

**Правила:**
- Разделы «Структура» и «Зависимости» — обязательны; «Точка входа / Заметки» — по
  необходимости.
- Для листовых пакетов допускается сокращённая форма (Назначение + таблица файлов).
- README — навигация и правила границ; подробные контракты — в Javadoc.
- README на русском; пути, имена типов и пакетов — как в коде.

## Код

- **Неизменяемость value objects.** Доменные модели — `record`; коллекции
  копируем при входе (`List.copyOf`).
- **Внедрение через конструктор.** Никакого field-injection; ядро не знает о DI.
- **Порты узкие.** Один порт — одна ответственность; имя отражает роль.
- **Адаптеры не торчат мимо порта.** Снаружи виден контракт, не реализация.
- **Специфику — в конфиг.** Никакого хардкода источников/приёмников/паттернов/
  политик в коде ядра.
- **Паттерны — RE2-совместимы** (`\b`, без lookaround/backref), чтобы работали на
  обоих движках `PatternEngine`.
- Язык кода, идентификаторов и комментариев — английский.

## Тесты

- **Изоляция от артефактов проекта.** Тесты **не** читают и не сверяются с
  `source/*`, `dataframe/*`, прод-`application.yml`. Иначе смена источника, поля
  выходного артефакта или одного значения конфига роняет тест — это запрещено.
- **Тест владеет данными.** Фикстуры синтетические и минимальные, создаются в
  `@TempDir` или `src/test/resources` (тестовый контур, tmp); конфиг/политику тест
  задаёт сам (inline доменные объекты или тест-конфиг).
- **Выход — во временные каталоги** (`@TempDir`), не в `dataframe/`.
- **Проверяем логику, не текущие данные.** Тест-корпус (case-table + golden)
  живёт в тест-ресурсах; кейсы изолированы — добавление одного не ломает другие.
- JUnit 5 + AssertJ; для case-table — параметризованные тесты (`@MethodSource`/CSV).
  Детали корпуса — [extraction.md](extraction.md#тест-корпус-и-изоляция-тестов).
